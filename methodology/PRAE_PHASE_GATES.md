# PRAE Phase Gates

> **Purpose**: Natural-language rules for the four phase gates — which file the AI writes, what it writes, how a human approves, and what happens after approval
> **Audience**: LLM (you)
> **Status**: Active (PRAE v1.0)
> **Last updated**: 2026-04-19

---

## 1. General Principles

A PRAE phase gate = a two-step mechanism of **AI analysis + human approval**.

- The AI generates `PHASE_GATE.md`: it lists the evidence, checks the gate conditions, and gives an advancement recommendation
- The human fills in the `APPROVED` field at the end of `PHASE_GATE.md`: this leaves an approval record
- After approval, you must advance through the official tooling: call `prae advance-phase` again, or run `tools/check_phase_gate.py --check-approved` + `tools/advance_phase.py`

**You (the AI) never advance a phase directly.** Even if every condition is met, you must stop and wait for the user's approval.

**Tool**: `tools/check_phase_gate.py` validates the structural legality of `PHASE_GATE.md` and the legality of the phase transition.

---

## 2. Common PHASE_GATE.md Structure

Every `PHASE_GATE.md` must contain the following sections (missing any one section means it is non-compliant):

```markdown
# Phase {from} → Phase {to} gate analysis

**Generated date**: YYYY-MM-DD
**Generated by**: AI (Analyst role)
**Target phase**: phase_NN_xxx

## 1. Current phase state
(List the id, type, and state of every track in track_registry.yaml)

## 2. Item-by-item gate condition check
- [x] Condition 1 (met)
- [ ] Condition 2 (not met)
...

## 3. Evidence summary
(1-2 sentences per key piece of evidence, linked to EXP_NNN.md or TRACK_LOG.md)

## 4. Risks and open items
(Risks that may affect the next phase, unresolved questions)

## 5. Recommendation
**Recommended action**: advance / do not advance yet
**Rationale**: ...

## 6. Pending human approval
APPROVED: <pending | yes | no>
APPROVER: <user name or identifier>
APPROVED_AT: <YYYY-MM-DD>
COMMENT: <optional, additional human notes>
```

## 3. Phase 0 → Phase 1 Gate

### 3.1 Entry Prerequisites

- The project has completed `PRAE_INIT.md` initialization (problem statement + component classification)
- `prae/track_registry.yaml` already lists all infrastructure tracks and research tracks
- Every infrastructure track has gone through at least the `EXPLORING` phase

### 3.2 The AI's Preparation Work

Switch to the **Analyst** role. Generate `prae/phases/phase_00_infra/PHASE_GATE.md` following these steps:

1. Read `track_registry.yaml` and filter for tracks with `type=infrastructure`
2. Check each infrastructure track:
   - Does `state` equal `LOCKED`?
   - Does the `locked_at` field exist?
   - Does the corresponding `src/infra_{name}_v{N}/` directory exist?
   - Does `contracts.yaml` exist?
   - Does MODULE_SPEC.md exist (proving PDAE M1-M3 has been completed)?
3. For each infrastructure track, read `TRACK_LOG.md` and confirm the selection conclusion and PDAE pass record
4. Fill in PHASE_GATE.md. **Section 2** must include the following conditions (check each one):
   ```
   - [ ] All tracks with type=infrastructure have state = LOCKED
   - [ ] Every LOCKED track has a contracts.yaml
   - [ ] Every LOCKED track has a MODULE_SPEC.md
   - [ ] Every LOCKED track has passed the PDAE M3 gate (recorded in TRACK_LOG.md)
   - [ ] check_contracts.py passes for all infrastructure contracts
   ```
5. Section 3 references the selection conclusion and PDAE pass evidence from each track's TRACK_LOG.md
6. Section 5 explicitly states "advance" / "do not advance yet"

### 3.3 Form of Human Approval

The user fills in Section 6 of `PHASE_GATE.md`:
```
APPROVED: yes
APPROVER: maintainer
APPROVED_AT: 2026-04-22
COMMENT: The data pipeline has passed the pre-launch two-person review and may enter Phase 1
```

### 3.4 Follow-up Actions After Approval

After the AI detects `APPROVED: yes`:

1. Run `python3 tools/check_phase_gate.py --project-dir . --check-approved` to verify that PHASE_GATE.md has `APPROVED: yes` filled in
2. Run `python3 tools/advance_phase.py --project-dir .` (or `prae advance-phase`)
3. The official tooling updates `track_registry.yaml.current_phase = phase_01_research`
4. The official tooling creates `prae/phases/phase_01_research/`, `PHASE_BRIEF.md`, and the research track `TRACK_LOG.md` files

### 3.5 Common Blockers

| Symptom | Cause | Handling |
|------|------|------|
| An infrastructure track is still EXPLORING | Selection not finalized or PDAE not passed | Return to Phase 0 and continue working on that track |
| contracts.yaml exists but check_contracts fails | A research track or test code violates the contract | Fix the violating code or fix the contract scope |
| MODULE_SPEC.md is missing | The PDAE process was not completed | Run PDAE M1-M3 to completion |
| Human approval `APPROVED: no` | The review did not pass | Read the COMMENT and reinforce as recommended |

## 4. Phase 1 → Phase 2 Gate

### 4.1 Entry Prerequisites

- Phase 0 → Phase 1 has been approved (current_phase = phase_01_research)
- Every research track has run at least one experiment during Phase 1
- There is enough evidence to judge which tracks deserve to enter the validation period

### 4.2 The AI's Preparation Work

Switch to the **Analyst** role and generate `prae/phases/phase_01_research/PHASE_GATE.md`:

1. Read `track_registry.yaml` and list all tracks with `type=research` and their state
2. For each research track:
   - Check whether `TRACK_LOG.md` already records ≥1 experiment
   - Check whether there is at least one `EXP_NNN.md` under `experiments/` with both `## Result` and `## Conclusion` filled in
   - Read each track's `evidence_summary`
3. Section 2 includes the following conditions:
   ```
   - [ ] ≥1 research track has state = ACTIVE (positive signal)
   - [ ] Every research track still in EXPLORING has a clear keep/drop recommendation (continue / KILL / merge)
   - [ ] Every ACTIVE track passes the Research Gate check
   ```
4. Section 3 writes an evidence summary for each ACTIVE track (1-2 sentences, linked to the key EXP)
5. Section 4 lists any systemic risks that have been found (e.g., the interface of some infrastructure dependency is not stable enough)
6. Section 5 recommends "advance" or "do not advance yet"

### 4.3 Form of Human Approval

Same as Phase 0 → 1: fill in the `APPROVED` field in Section 6.

### 4.4 Follow-up Actions After Approval

1. Run `python3 tools/check_phase_gate.py --project-dir . --check-approved`
2. For research tracks whose terminal state needs to be recorded, first run `python3 tools/update_track_state.py ...` (or `prae update-track-state ...`). **Note: a track still in EXPLORING cannot move directly to a terminal state; if it needs to be stopped, you must first record one experiment to bring it to ACTIVE, then mark it KILLED.**
3. Run `python3 tools/advance_phase.py --project-dir .` (or `prae advance-phase`)
4. The official tooling updates `track_registry.yaml.current_phase = phase_02_validation`
5. The official tooling creates `prae/phases/phase_02_validation/`, `PHASE_BRIEF.md`, and the validation-period track logs

### 4.5 Common Blockers

| Symptom | Cause | Handling |
|------|------|------|
| All research tracks are still EXPLORING | Insufficient signal, too few experiments | Return to Phase 1 and continue running experiments |
| Evidence summaries contradict each other | The experiment design is flawed, results are not reproducible | Return to Phase 1 and redesign the experiment |
| The infrastructure that a research track's depends_on points to is still EXPLORING | Phase 0 was not closed out | Return to Phase 0 and complete it |

## 5. Phase 2 → Phase 3 Gate

### 5.1 Entry Prerequisites

- Every ACTIVE research track has run rigorous validation experiments during the validation period
- Every track has a clear conclusion recommendation (KILLED / MERGED / GRADUATED)

### 5.2 The AI's Preparation Work

Generate `prae/phases/phase_02_validation/PHASE_GATE.md`:

1. Read all research tracks in `track_registry.yaml`
2. For each track, judge the terminal-state recommendation:
   - KILLED: the evidence falsifies the hypothesis
   - MERGED: the evidence highly overlaps with or complements another track, and merging is more reasonable
   - GRADUATED: the evidence is sufficient and the track deserves to be engineered
3. Section 2 includes:
   ```
   - [ ] Every formerly ACTIVE research track has a clear terminal-state recommendation
   - [ ] At least one track is judged GRADUATED (otherwise consider KILLing the whole project)
   - [ ] Every GRADUATED candidate passes the Research Gate and the Contracts Gate
   - [ ] Every GRADUATED candidate has a final conclusion paragraph in `TRACK_LOG.md`
   ```
4. Section 3 provides evidence links for each track's terminal-state recommendation
5. Section 4 lists open items affecting the conclusion (e.g., external data delay issues, effects requiring long-term observation)
6. Section 5 recommends "advance to Phase 3" or "return to Phase 2 to gather more evidence"

### 5.3 Form of Human Approval

When filling in Section 6, the human should additionally explicitly approve or reject each track's terminal-state recommendation. Recommended format:

```
APPROVED: yes
APPROVER: maintainer
APPROVED_AT: 2026-05-10
COMMENT:
  - research_strategy_momentum: GRADUATED approved
  - research_strategy_reversal: KILLED approved
  - research_strategy_mean_revert: MERGED into research_strategy_momentum approved
```

### 5.4 Follow-up Actions After Approval

1. Run `python3 tools/check_phase_gate.py --project-dir . --check-approved`
2. For each research track, call `python3 tools/update_track_state.py ...` (or `prae update-track-state ...`) according to its terminal state:
   - `KILLED / MERGED / GRADUATED` are written into the registry by the official tooling
   - The official tooling fills in `concluded_at` at the same time
   - For a `MERGED` track, the official tooling enforces that `merged_into` is set
3. Run `python3 tools/advance_phase.py --project-dir .` (or `prae advance-phase`)
4. The official tooling updates `current_phase: phase_03_conclusion`
5. The official tooling creates `prae/phases/phase_03_conclusion/PHASE_BRIEF.md`

### 5.5 Common Blockers

| Symptom | Cause | Handling |
|------|------|------|
| No track is judged GRADUATED | The research problem is harder than expected | If the project has failed overall, recommend that the human decide whether to terminate the project |
| A track has insufficient evidence but was subjectively judged KILLED | Too few validation experiments | Run more rigorous validation experiments |
| MERGED, but the target track is not yet ready to receive it | The receiving track's hypothesis does not yet include the merged content | First extend the hypothesis of the target track |

## 6. End of Phase 3 (Project Wrap-up)

### 6.1 Entry Prerequisites

- Phase 2 → 3 has been approved
- Every research track has reached a terminal state

### 6.2 The AI's Preparation Work

This phase has no phase-transition gate (Phase 3 is the endpoint). The AI generates `prae/phases/phase_03_conclusion/CONCLUSION.md`, organizing:

- The project's final conclusion (Was the core problem solved? Which hypotheses were confirmed?)
- A disposition list for each track
- The location (link) of the PDAE project for each graduated track
- Unresolved problems / potential new research projects to open later

### 6.3 Human Decision

The human fills in the structured fields at the end of `CONCLUSION.md`:

```text
APPROVED: <pending | yes | no>
DECISION: <ARCHIVED | GRADUATED_TO_PDAE | CONTINUE>
APPROVER: <user name or identifier>
APPROVED_AT: <YYYY-MM-DD>
COMMENT: <optional, additional human notes>
```

The meaning of `DECISION` is as follows:

- `ARCHIVED`: the entire PRAE project is archived; no PDAE is started
- `GRADUATED_TO_PDAE`: at least one track enters a PDAE engineering project (the project is not archived and remains traceable)
- `CONTINUE`: a new problem was discovered and you may consider opening another Phase 1 (rare, requires a strong rationale)

Execution rules:

- `ARCHIVED` / `GRADUATED_TO_PDAE`: run `python3 tools/finalize_project.py --project-dir .`
- `CONTINUE`: run `python3 tools/reopen_project.py --project-dir .`, archive this cycle's `phase_01/02/03` directories as a whole into `prae/history/cycle_N/phases/`, and reopen to `phase_01_research` with `current_cycle = N+1`

### 6.4 AI Actions After GRADUATED_TO_PDAE

Switch to the **Executor** role:

1. For each GRADUATED track, switch to the PDAE repository
2. Create a new PDAE engineering project following `PDAE_QUICKSTART.md`
3. Use the research track code as the factual input for `scout_m1`
4. Start the PDAE control thread and run the full M1-M3
5. After the PDAE project is created, append a link to the PDAE project path in the PRAE project's `TRACK_LOG.md`

## 7. Phase Reversal / Rollback

**By default, phase transitions are one-way.** Once you have entered Phase 2, you cannot return to Phase 1.

**Exception**: when a serious problem is discovered (e.g., a fundamental flaw in an infrastructure contract), the user may manually:

1. Write `current_phase_override: phase_00_infra` at the top of `track_registry.yaml` and add a comment
2. From this moment on, phase-aware tools interpret the current phase as `phase_00_infra`; the regular `generate/check/advance-phase` should no longer be used
3. After receiving the instruction, the AI **stops advancing** and only handles the new v2 infrastructure tracks (e.g., `prae add-track` / `prae new-track` / `prae new-exp` / `prae lock-infra`)
4. Research tracks remain in their original phase directories but are no longer advanced until the override is removed by the user
5. After the v2 track is LOCKED, the user removes `current_phase_override`, then decides whether to continue the validation period or re-explore

This path does not go through the regular PHASE_GATE mechanism; it is explicitly operated by the user. The AI should not initiate a phase rollback on its own.

---

## 8. Key Rules Recap

1. Every phase transition must produce a `PHASE_GATE.md` with a fixed structure (6 sections)
2. `PHASE_GATE.md` must indicate the current `research cycle (cycle_N)` and be consistent with `track_registry.yaml`
3. The AI can only recommend; it cannot advance a phase directly
4. `APPROVED: yes` is the only legal advancement signal
5. After approval, the `current_phase` in `track_registry.yaml` must be updated through the official tooling
6. Unmet gate conditions cannot be skipped or falsely checked
7. All PHASE_GATE.md files are kept as permanent records after approval and are not deleted
